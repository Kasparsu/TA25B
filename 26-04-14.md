### 2. 🏷️ Git Tags

The first new topic was **Git tags** — permanent labels that pin a specific commit by name, most commonly used to mark software releases.

**Why tags vs branches:**
A branch pointer moves forward with every new commit. A tag stays where it was set forever. If you tag a commit as `v1.0.0`, anyone checking out that tag a year later gets exactly that code — no matter how many commits have been added since.

**Two types of tags:**

| Type | Command | Notes |
|------|---------|-------|
| Lightweight | `git tag v1.0.0` | Just a pointer to a commit — no extra metadata |
| Annotated | `git tag -a v1.0.0 -m "Release 1.0.0"` | Includes tagger, date and message — preferred for releases |

**Useful tag commands:**
```bash
git tag                           # list all existing tags
git tag v1.0.0                    # lightweight tag at current HEAD
git tag -a v1.0.0 -m "message"   # annotated tag
git push origin v1.0.0            # push one specific tag
git push origin --tags            # push all tags at once
git checkout v1.0.0               # check out the exact state at that tag
```

> ⚠️ Tags are **not pushed automatically** with a normal `git push`. You must push them explicitly.

### 3. 🔢 Semantic Versioning (SemVer)

Tags are most useful when paired with **Semantic Versioning** — the widely adopted standard for naming software releases.

A SemVer version looks like **`MAJOR.MINOR.PATCH`** — for example `2.4.1`

| Number | When it changes | Example |
|--------|-----------------|---------|
| **MAJOR** | Breaking change — existing code may stop working | Renamed a function or changed an API contract |
| **MINOR** | New features added in a backward-compatible way | Added a new function, nothing old breaks |
| **PATCH** | Bug fixes only — nothing new, nothing broken | Fixed a typo causing a crash |

**Practical advice from the session:**
- `0.x.y` = still in development, not publicly stable — anything can change
- `1.0.0` = first stable public release — from here the rules apply strictly
- When a `.x.0` (minor) version comes out fresh, wait a little — there are often teething bugs patched quickly in `.x.1` and `.x.2`
- Don't rush to update to `x.0.0` (major) the moment it drops — let the community find and fix the sharp edges first
- Large programs like VS Code (currently `1.88.134`) may have patch numbers in the hundreds — that's normal, it just means lots of small fixes over time

### 4. ✍️ How to Write Good Commit Messages

A substantial section was dedicated to **commit message best practices** — one of the most visible professional habits a developer can build.

**Why it matters:**
Your commit log is a searchable project diary. With good messages you can `git log --grep="fix"` and find all bug fixes instantly. A year later — or when a new team member joins — a log of `"fix"`, `"asdf"`, `"changes"`, `"stuff"` tells no one anything. Three minutes spent writing a clear message saves hours of archaeology later.

**The rules covered:**

**1. Imperative mood in the subject line (käskiv kõneviis)**
Complete the sentence: *"If applied, this commit will…"*
- ✅ `Add pagination to character list`
- ✅ `Fix null error when search input is empty`
- ❌ `Added pagination` (past tense)
- ❌ `Adding pagination` (continuous)

This convention comes from Git itself — all of Git's own auto-generated messages use imperative mood. It has become the universal standard.

**2. Keep the subject line under 72 characters**
Many tools (GitHub UI, `git log --oneline`) truncate or wrap at around 72 characters. VS Code extensions can display a ruler at this length.

**3. Do not end the subject line with a period**
It's a title, not a sentence.

**4. Use the body (separated by a blank line) for the *why*, not the *what***
The diff already shows *what* changed. The body should explain *why* — especially useful for non-obvious decisions, bug workarounds, or linking to tickets/issues.

```
Fix empty search triggering API call

The R&M API returns all characters when name="" is sent,
which caused an unintentional full reload on every keystroke.
Added an early return when search value is empty.
```

**5. One logical change per commit**
Don't mix "add feature X" with "fix bug in Y" in the same commit.

**6. Commit messages are searchable**
`git log --grep="fix" --author="Kaspar"` — searchable by content, author, date. Good subject lines make this powerful.

### 5. 📦 `git stash` — Saving Work in Progress

A new command introduced this session: **`git stash`** — a way to temporarily shelve uncommitted changes without making a commit, so you can switch context and come back later.

**The scenario it solves:**
You're in the middle of working on something. You haven't committed yet because it's not done. But now you need to switch branches (maybe to fix an urgent bug, or to pull new changes). Git won't let you switch with uncommitted changes that would conflict. You don't want to commit a half-finished mess either. Stash it.

```bash
git stash              # save all uncommitted changes to the stash
git stash pop          # restore the most recently stashed changes
git stash apply        # restore without removing from stash
git stash list         # see all stashes
git stash drop         # delete the top stash entry
```

**What `stash` saves:**
All modified tracked files and staged changes are packed away into a hidden "stack". Your working directory returns to the state of the last commit. After switching branches and doing what you needed, `git stash pop` brings your work-in-progress back.

> 💡 Think of stash like a clipboard for your uncommitted work — cut, do something else, paste back.

### 6. 🗜️ `git commit --fixup` & `git rebase --autosquash`

The most advanced topic of the session — **autosquash**, a way to automatically clean up fix commits when you run `git rebase -i`.

**The problem it solves:**
You made several commits. Later you spot a mistake in an earlier commit. You don't want a messy "fix typo in commit 3" commit polluting your log. But you also don't want to manually open `git rebase -i` and drag commits around.

**Step 1 — Make a fixup commit with a special flag:**
```bash
git add <fixed-file>
git commit --fixup=<hash-of-commit-you-are-fixing>
```
This creates a commit with the message `fixup! <original commit message>` — Git recognises this prefix automatically.

**Step 2 — Run rebase with autosquash:**
```bash
git rebase -i --autosquash HEAD~<n>
```
Git automatically moves your `fixup!` commit directly below the commit it targets, and pre-marks it as `fixup` in the interactive editor. You can then simply save and close — no manual rearranging needed.

**What the result looks like:**
The fixup commit disappears from history, its changes silently merged into the commit it corrects. The log is clean, as if you had written the original commit correctly the first time.

**The same limitation as always:**
Only use this before pushing — rewriting already-pushed history causes problems for everyone else on the branch.

### 7. ↩️ `git reset` — Undoing Commits

The session also covered **`git reset`** with its three flavours for rolling back commits:

```bash
git reset --soft HEAD~1    # undo last commit, keep changes staged
git reset HEAD~1           # undo last commit, unstage changes (keep files)
git reset --hard HEAD~1    # undo last commit AND delete the changes
```

| Flag | Branch pointer | Staging area | Working files |
|------|---------------|--------------|---------------|
| `--soft` | moves back | unchanged (staged) | unchanged |
| `--mixed` (default) | moves back | cleared | unchanged |
| `--hard` | moves back | cleared | **deleted** |

You can also target a specific commit hash instead of `HEAD~1`:
```bash
git reset --hard <hash>
```

**"Git never truly deletes anything":**
Even after `--hard`, the discarded commits still float in Git's internal storage for a while, accessible via `git reflog`. If you accidentally reset too far, `git reflog` lists every position HEAD has pointed to — find the hash and `git checkout <hash>` to rescue it.

### 8. 🕰️ `git checkout <hash>` & Detached HEAD

Covered in conjunction with `git reset` — using `git checkout` with a commit hash to navigate back to any point in history:

```bash
git log --oneline          # find the commit hash
git checkout abc1234       # travel to that exact commit
```

This enters **detached HEAD** state — you're not on any branch, just floating at a specific commit. You can read files, run the code, inspect the state — but any new commits made here won't belong to any branch. Use `git checkout main` (or `git switch main`) to return.

**`git checkout -`** — mentioned as a handy shortcut:
```bash
git checkout -    # switch back to the previous branch you were on
```
This works exactly like `cd -` in the terminal — "go back to where I just was."
