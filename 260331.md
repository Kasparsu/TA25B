### 2. 📥 `git clone` — The School-to-Home Workflow

A large practical portion was dedicated to demonstrating the real-world **multi-machine workflow**. The instructor simulated cloning a repo into a second folder (acting as a "home computer") to show how students can continue work anywhere.

```bash
git clone <url>              # clone the repo (creates a folder named after the repo)
git clone <url> my-folder    # clone with a custom folder name
```

**Key points:**
- After cloning, Git already knows the remote — no need to re-add origin
- `git log` shows the full identical history
- `git status` should always be the first command when sitting at a new machine
- **Never make local changes before pulling** — this risks merge conflicts

**The daily loop:**
```
Morning at school   → git pull → code → git add → git commit → git push
Evening at home     → git pull → code → git add → git commit → git push
Next morning        → git pull → continue...
```

### 3. 🔍 `git fetch` vs `git pull` (Revisited)

The distinction was reinforced with live examples:

| Command | What it does |
|---------|--------------|
| `git fetch` | Downloads info about remote changes — **does not touch your local files** |
| `git pull` | Fetches **and** merges remote changes into your current branch |
| `git log --all` | Shows all commits including remote ones after a fetch |

The instructor showed that after `git fetch`, `git log --all` reveals the remote's newer commits, while the local branch hasn't moved yet — making the distinction very visual.

### 4. 🌿 Git Branches — Deep Dive

The second half of the session focused heavily on **branching** — the core workflow of professional software development.

**Why branches exist:**
- `main` should always contain stable, working code — ideally what's deployed live
- New features and experiments go into their own branches to avoid breaking `main`
- In real teams: nobody pushes directly to `main`; everything goes through feature branches and review

**Real-world branch structure explained:**
```
main          ← stable, deployed code (protected)
develop       ← where tested features accumulate
feature/xyz   ← individual developer's branch for a specific task
```

**Branch commands covered:**

| Command | What it does |
|---------|--------------|
| `git branch` | Lists all local branches; current branch marked with `*` |
| `git branch <name>` | Creates a new branch |
| `git checkout <name>` | Switches to an existing branch |
| `git checkout -b <name>` | Creates **and** switches to a new branch in one step |
| `git merge <branch>` | Merges the named branch into the current branch |
| `git push -u origin <branch>` | Pushes a local branch to GitHub for the first time |

**Merge workflow demonstrated:**
1. Make commits on a feature branch
2. Switch back to `main` with `git checkout main`
3. Run `git merge <feature-branch>` to bring the changes in
4. After merge, the feature branch can be deleted (it's served its purpose)

![Git merge meme](https://i.imgur.com/gPWmEZX.gif)  
*"git merge in action"*

**Pushing a branch to GitHub:**
- On the first push of a new branch, Git will complain and suggest the full command with `--set-upstream`
- Git helpfully prints the exact command to run — just copy and paste it
- After that first push, `git push` alone works for that branch

**Viewing branches on GitHub:**
- After pushing, the branch appears on GitHub — click the branch selector dropdown to see it
- Pull requests (merging branches via GitHub) were mentioned as a topic for a future session
