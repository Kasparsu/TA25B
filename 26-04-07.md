### 2. 💥 Merge Conflicts — What They Are & How to Resolve Them

The main practical topic for the first half of the session was **merge conflicts** — deliberately creating one and then resolving it in different ways.

**When do conflicts happen:**
When two branches have edited the **same lines** of the same file differently, Git cannot decide which version is correct on its own. Instead of merging automatically, it stops and marks the conflict in the file, asking you to decide.

**What a conflict looks like in the file:**
Git inserts conflict markers directly into the affected file:
```
<<<<<<< HEAD
your current branch's version of the line
=======
the incoming branch's version of the line
>>>>>>> feature-branch
```
- `<<<<<<< HEAD` to `=======` — what your current branch has
- `=======` to `>>>>>>>` — what the incoming branch has
- You must delete the markers and choose the correct final content

**Resolving manually:**
1. Open the conflicted file
2. Find the conflict markers
3. Delete everything you don't want, including all three marker lines
4. Save the file
5. `git add <file>` — mark as resolved
6. `git commit` — complete the merge

**Resolving in VS Code:**
VS Code colour-highlights the conflict sections and provides four clickable buttons:
- **Accept Current Change** — keep only your branch's version (removes the incoming)
- **Accept Incoming Change** — keep only the other branch's version (removes yours)
- **Accept Both Changes** — keep both versions stacked on top of each other
- **Compare Changes** — opens a side-by-side diff view

The instructor noted that "Accept Both" is sometimes the right choice when both versions contain code that belongs in the final result, but in real projects the conflicts are often far more complex than a simple button press can solve — especially when two developers have changed the same function in genuinely different ways.

**After resolving:**
- Save the file
- `git add <file>` to stage the resolved version
- `git commit` — Git will auto-suggest a merge commit message

### 3. ⚡ Git Aliases & `.zshrc`

After the conflict exercise the instructor introduced **Git aliases** — custom terminal shortcuts for long commands.

**Why bother:**
Typing `git log --oneline --all --graph` repeatedly is tedious. An alias lets you define a short name for any command, so you can just type `gl` (or whatever you choose) instead.

**How to set them up:**
```bash
# Open the shell config file
code ~/.zshrc

# Add aliases at the bottom, for example:
alias gl="git log --oneline --all --graph"
alias gs="git status"
```

After saving, run `source ~/.zshrc` to load the changes immediately without restarting the terminal. The instructor demonstrated this live — noting that typing `~` on an Estonian keyboard requires pressing shift + the tilde key, then a space.

> 💡 Making your workflow faster by eliminating repetitive typing is a hallmark of a good programmer mindset.

### 4. 🔀 `git rebase` — A Cleaner Alternative to Merge

The second half of the session introduced **`git rebase`** — covering both why it exists and when to use it instead of `git merge`.

**The key difference:**

| `git merge` | `git rebase` |
|-------------|--------------|
| Creates a new merge commit | Rewrites commits so they appear after the target |
| History shows branching | History looks linear — as if you worked on main all along |
| Safe for shared branches | Should **only** be used on your own local branches |

![Merge vs Rebase diagram](https://opensource.com/sites/default/files/uploads/merge-commit-vs-rebase.png)

**How rebase works:**
Instead of creating a merge commit, rebase takes your feature branch's commits and replays them one by one on top of the latest version of the target branch. The result is a perfectly linear commit history — as if you had started your feature branch from the current tip of main all along.

**Basic usage:**
```bash
# While on your feature branch:
git rebase main
# Replays your commits on top of the latest main
```

**When rebase hits conflicts:**
Rebase can also create conflicts — the same rules apply, but instead of one big merge conflict, you resolve them commit by commit. After resolving each one:
```bash
git add <file>
git rebase --continue
```

**The golden rule:**
Never rebase a branch that others are also working from — rebase rewrites commit history, which will cause serious problems for anyone who already has the old commits.

### 5. ✏️ `git commit --amend` — Fixing the Last Commit

The session also covered **`git commit --amend`** as a way to fix the most recent commit without creating a new one.

**What it can do:**
- Fix a typo in a commit message
- Add a file you forgot to include in the last commit
- Combine a small "oops" fix into the commit it belongs to

**Usage:**
```bash
# Fix only the commit message:
git commit --amend -m "Corrected message here"

# Add a forgotten file:
git add forgotten-file.txt
git commit --amend --no-edit   # keeps the original message
```

**Important caveat:**
`--amend` rewrites the last commit and changes its hash. If you've already pushed that commit to GitHub, you'd need to force-push, which can cause problems for anyone else working on the same branch. Use `--amend` only on commits that **haven't been pushed yet**.

### 6. 🗜️ `git rebase -i` — Interactive Rebase (Squashing Commits)

The session closed with **interactive rebase** — a way to clean up multiple commits before pushing them.

**What it lets you do:**
- **Squash** several small commits into one clean, meaningful commit
- **Reword** commit messages
- **Drop** commits you don't want
- **Reorder** commits

**Usage:**
```bash
# Interactively rebase the last N commits:
git rebase -i HEAD~4   # opens editor showing last 4 commits
```

In the editor, each commit is listed with `pick` in front. Change `pick` to:
- `squash` (or `s`) — combine with the commit above
- `reword` (or `r`) — edit the message
- `drop` (or `d`) — delete the commit

The instructor demonstrated squashing several commits (including one with a deliberate typo) down into a single clean commit. Viewing `git log` afterwards shows a much tidier history — one commit instead of four messy intermediate ones.

**Conflicts during interactive rebase:**
When rewriting commits that touch the same lines, a conflict can appear during `git rebase -i` as well. The resolution process is the same: fix it, `git add`, then `git rebase --continue`.
