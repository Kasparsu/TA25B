## 🗂️ Topics Covered

### 1. 🗃️ Git — Why It Matters
The session opened with motivation: Git is not optional. The instructor stressed that without knowing Git, students won't be able to work in software professionally. Future assignments will also be submitted via Git, and teachers may request that all code-related work goes through it. Git keeps your code safe even if your laptop is stolen, breaks, or an office burns down.

The analogy used: just like saving a game before trying something risky, Git lets you save a known-good state and experiment freely — no more `project_final_v2_REAL_final.zip` folders.

---

### 2. ⚙️ Git Configuration (First-Time Setup)
Before using Git, a few one-time configurations need to be done — these are stored globally on the machine and tied to the user's account:

```bash
git config --global user.name "Your Full Name"
git config --global user.email "your@email.com"
git config --global core.editor "code --wait"
```

- The **name and email** identify who made each commit — important for grading and real-world accountability
- The **default editor** was set to VS Code so that when Git needs a message, it opens a familiar interface
- Students were reminded to be logged in with their own accounts, not shared school accounts, so configs are saved correctly

---

### 3. 🏗️ Starting a Repository — `git init`
- Created a new project folder (e.g. `ta25b`) and navigated into it
- Ran `git init` to initialise the repository — this creates a hidden `.git` folder
- The `.git` folder is where Git stores all history; deleting it with `rm -rf .git` removes Git tracking entirely
- VS Code typically hides `.git` by default, but it can be viewed with `ls -a` in the terminal

---

### 4. 📸 The Staging & Commit Workflow

The core Git loop was introduced step by step:

```bash
git status        # See what's going on
git add .         # Stage all changes ("put on stage")
git commit        # Save the staged snapshot with a message
```

- **`git status`** — described as "very important and informative"; always shows the current state and suggests next commands
- **`git add`** — stages files (the "stage" metaphor: only what's on stage gets saved)
- **`git commit`** — saves a permanent snapshot; every commit **must** have a message
  - Using `git commit` alone opens the configured editor (VS Code) for a longer message
  - Lines starting with `#` are comments and are ignored
- The value of meaningful commit messages was emphasised — they help you (and others) understand what changed and why

### 5. 🌐 GitHub — Remote Repositories
- Students logged into **GitHub** and created a new remote repository
- **GitHub** was introduced as one of the most well-known Git hosting platforms (others exist too)
- The local repo was linked to the remote:

```bash
git remote add origin <repository-url>
git branch -M main
git push -u origin main
```

- On first push, Git may ask for browser-based authentication
- After pushing, the commit is visible on GitHub — including who made it, when, and what changed (the diff view with `+` for added lines)
- The instructor showed the **commit hash** (a unique ID for every commit) visible in the GitHub UI

---

### 6. 🔁 Subsequent Commits & Push
Once the repo is set up, the ongoing workflow is simpler:

```bash
git add .
git commit -m "describe what changed"
git push
```

- Students practiced making a change, committing, pushing, and verifying it appeared on GitHub with a refresh
- `git log` was briefly shown — it lists all commits with their hashes, author, and timestamp
